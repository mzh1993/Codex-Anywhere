# Product Decision Baseline

> Internal product correction note. This document is not for public marketing copy.

## 文档定位

- 这是一份内部产品纠偏决策书。
- 它不用于描述“当前已经做了什么”。
- 它用于裁决“接下来什么方向可以做，什么方向不能做”。
- 今后所有桥接器交互改动，都应先经过这份文档的判断。
- 具体 V1 协议只在 `docs/feishu-codex-runner-v1.md` 维护，不在这里重复展开。

解释：
这不是经验总结，也不是 issue 列表，而是一份裁判标准。它的作用不是帮助团队为现状找理由，而是帮助团队在后续设计中更早发现偏航。

## 一句话裁决

- 我们做的是一个尽量贴近原生 Codex 交互的远程桥。
- 我们不做一个重新封装 Codex 语义的 IM 机器人。

解释：
桥层的职责是把 Codex 的原生协作带到远程入口，而不是自己再发明一套新的主交互语义。桥可以有内部控制平面，但不应成为用户主要感知到的产品。

## 第一性原则

- 渠道不决定执行语义。
- 用户只学一套通用用户语言。
- 默认输入优先按原生 Codex 会话理解。
- 桥接命令只能是兜底入口，不能成为主交互路径。
- 复杂性可以留在桥内部，不能外溢给用户。

解释：
远程入口天然会带来鉴权、审批、恢复、审计等复杂性，但这些复杂性应由桥层吸收，而不应变成用户必须学习的新命令、新状态和新阅读负担。

## 新交互基线

- 用户几乎只做两件事：发自然语言、看 Codex 回复。
- 默认续写、恢复、审批都优先走自然语言。
- Codex 主输出优先，桥层提示退居次位。
- 桥层只在少数必要场景主动浮出：审批、边界拒绝、恢复提示、repo-owned control-plane。
- 即使出现歧义，也优先自然语言澄清，而不是命令化交互。

解释：
桥层不应持续存在于用户视野中。正常情况下，用户应感知到自己正在远程继续一个 Codex 会话；只有在审批、边界拒绝、恢复提示、repo-owned control-plane 这类必要时刻，bridge 才短暂浮出。

## 桥层职责边界

- 必须留在桥层的：鉴权、安全边界、审批、恢复、审计、渠道适配。
- 必须还给 Codex 的：会话连续性、自然语言续写、普通任务推进、原生交互节奏。
- 桥层负责守门，不负责重新定义会话。
- 桥层像安全内核，不像主交互应用。

解释：
凡是远程入口天然新增的责任，应留在桥层；凡是本来就属于 Codex 会话本身的能力，应尽量还给 Codex。桥内部可以严格，但桥外部应尽量轻。

## 对象边界

- `codex task` 是主产品对象。
- `bridge action` 是边界对象，不是主工作对象。
- `codex task` 负责工作连续性；`bridge action` 负责本仓库宿主控制面的一次性闭环。
- 两者共享同一个对话入口，但不共享执行语义和连续性状态。

解释：
系统内部必须承认，不是所有动作都属于 Codex 会话本身。普通工作内容属于 Codex；而桥接器自己的宿主控制面动作，本质上属于 bridge。只有先把对象边界切开，后续 owner、审批、恢复和状态表达才能稳定。

## 宿主控制面动作裁决

- 凡是本仓库专属宿主控制面动作，都由 bridge 直接拥有并执行。
- 这类动作按风险进入 bridge 自己的治理：只读查询可直接执行，变更类动作先审批后执行。
- 这类动作不再透传给 Codex 去“想办法执行”。
- V1 仅覆盖本仓库专属控制面：bridge/gateway 相关 service、健康检查、诊断、安装类动作。
- 不扩展为任意宿主机运维面板。

解释：
“重启 bridge 自己”这类动作不是普通 Codex 任务语义，而是桥接器宿主控制面语义。若仍把这类动作交给 Codex，在沙箱里它有时会真正执行，有时会退化成解释文本，结果不可预测。对这类动作，bridge 必须拥有直接执行权；区别只在于读类可直执，变更类先审批。

## 连续性裁决

- `bridge action` 不占用当前 `codex task` 的 session。
- `bridge action` 不污染当前 `codex task` 的 summary、next steps、cwd、恢复状态。
- 完成一个 `bridge action` 后，用户仍应无缝回到原来的 Codex 工作连续性。
- 控制面动作与工作任务可以同处一个私聊，但不能互相吞并。

解释：
同一私聊只说明“入口相同”，不说明“对象相同”。如果把宿主控制面动作塞进 Codex task，会把工作上下文弄脏；如果把 Codex task 反过来吞进 bridge action，又会让桥越权。正确做法是共享入口、分离连续性。

## Owner 路由新增裁决

- 默认 `owner=codex`。
- 只有 bridge 明确拥有的边界状态，才切到 bridge owner。
- 普通自然语言优先让路给 Codex。
- bridge 只接住自己明确拥有语义的动作与审批回复。

解释：
bridge 不应因为“自己也能理解一句话”就抢走语义。判断顺序应是：先看当前 owner，再看该动作是否天然属于 bridge。凡是不属于 bridge 明确职责范围的输入，默认透传给 Codex。

## 无感交互裁决

- 自然语言永远是主路径。
- slash 命令永远只是兜底入口。
- bridge 回复必须比 Codex 更短、更少、更晚出现。
- 除排障场景外，不主动暴露 task、run、token、owner 等内部对象。

解释：
桥层的优秀不是“更会说”，而是“更少被感知到”。用户应主要感知自己在和远程 Codex 协作，而不是在学习一个新的桥接器产品。

## 必须推翻的典型错误行为

- 不应再把 `/codex continue <prompt>` 当成主续写或主恢复路径。
- 不应再让桥层自定义命令成为日常交互核心。
- 不应再外露 `item.completed` 这类内部事件流。
- 不应再把 task、run、session、token 等内部概念大量压给用户。
- 不应再让桥层状态播报盖过 Codex 主回复。
- 不应再把歧义处理降级成命令式人工操作。

解释：
这些行为的问题不只是体验差，而是方向错。它们共同导致桥层越权接管了原生 Codex 语义，把本应留在内部的复杂性外溢成用户负担。

## 必须建立的新默认行为

- 默认最小状态表达。
- 默认优先展示任务内容，而不是桥层过程。

解释：
新的默认行为不是增加“更聪明的猜测”，而是把主路径重新交还给原生 Codex 会话。桥层仍然可以保留严格协议，但用户不应因此承担额外学习成本。

## 外部样本裁决

- 借鉴 `Johnixr/claude-code-wechat-channel` 的薄桥思路，不借鉴其相对弱化的安全边界。
- 借鉴 `op7418/Claude-to-IM-skill` 的内部工程分层，不借鉴其面向用户暴露过多桥层命令。
- 借鉴 OpenClaw 官方 channel/plugin contract，不接受其反向定义产品交互语义。
- 借鉴 `CodexBridge` 的 session-first 思路，不把它直接当作 IM 产品模板。
- 我们的方向不是复制任一现成项目，而是组合出更适合自己的结构。

解释：
外部样本的价值在于帮助我们看清不同层面的正确做法，而不是提供一个可以整体照搬的答案。我们的目标是：用户表面像薄桥，系统底座像受控执行平台。

## 后续开发裁判问题

- 这是否让渠道开始决定执行语义？
- 这是否把桥内部复杂性外溢给用户？
- 这是否削弱了原生 Codex 会话感？
- 这是否让桥接命令重新成为主路径？
- 这是否错误地把宿主控制面动作继续留在 Codex prompt 平面？
- 这是否错误地让 `bridge action` 污染了 `codex task` 的连续性？
- 这是否真的解决了用户问题，而不是仅仅缓解实现现状？
- 这是否在没有必要的情况下增加了阅读负担？

解释：
这组问题的作用不是帮助团队解释为什么可以做，而是优先帮助团队发现为什么不该做。只要其中任一问题明显指向错误方向，就应默认停止推进或重写方案。

## 第一原则检查清单

使用方式：

- 任何新的交互设计、协议变更、bridge 路由调整、审批语义调整、恢复逻辑调整，在进入实现前都先逐条过一遍。
- 任意一条答案如果明显是“否”，默认不应继续实现；先改方案，再推进。
- 这份清单不是补充讨论材料，而是实际裁决入口。

### A. 执行语义检查

- 这次改动是否仍然由状态机、审批协议、隔离边界、审计机制定义执行语义，而不是由消息样式、渠道特性或文案格式直接决定？
- 这次改动是否仍然允许 Feishu 之外的入口复用同一套执行语义，而不需要重写核心规则？
- 若请求只是 discussion / explanation / doc 语义，方案是否只是避免误判为真实执行，而不是借机放松 `host_secret`、`isolation_boundary` 等硬边界？

### B. 主对象检查

- 这次改动处理的主对象是否清楚：普通连续工作属于 `codex task`，仓库自有宿主控制面一次性动作属于 `bridge action`？
- 若一句话同时带有普通工作语义和控制面语义，方案是否默认回落到 `codex task`，而不是让 bridge 抢走混合语义？

### C. 审批边界检查

- 高风险动作是否仍进入明确审批协议，而不是混入普通对话流被隐式放行？
- 审批是否仍然是单次、run-scoped 的：批准的是下一次获批 run，而不是永久放开整段会话？
- 审批记录若已持久化 `approvalGrant`，run 真正启动前是否仍会复核当前边界；若边界已变化，是否保持原审批未消费而不是带病启动？
- 若后续把 `approvalGrant` 继续收口为更强的 typed grant，它是否仍只服务当前 `codex task` 的下一次获批 run，而不是一开始就做成跨对象通用票据？
- 若 `bridge action` 暂不复用 typed grant，方案是否仍保持 bridge 更薄，而不是为了抽象统一提前把控制面审批做厚？
- `approve_with_tail` 一类补充要求是否被限制在“不能放大边界”的范围内；若新增 bridge-owned 或 deny 边界，是否保持原审批未消费？
- run 真正启动失败时，approval token 是否仍会被保留，而不是提前吞掉？
- 同一 approval token 在成功启动一次后，是否会被明确标记为已消费，避免因删除滞后或状态漂移被重复启动？

### D. 连续性检查

- `bridge action` 是否仍不占用、不污染当前 `codex task` 的 `session / summary / next steps / cwd / recovery state`？
- 完成、拒绝、失败一个 `bridge action` 后，用户是否仍能自然回到原有 Codex 连续性？

### E. 用户感知检查

- 用户是否仍然主要做两件事：发自然语言、看 Codex 回复？
- slash 命令是否仍只是兜底入口，而不是主路径？
- bridge 是否仍只在审批、边界拒绝、恢复提示、repo-owned control-plane 这类必要时刻短暂浮出，而不是持续占据用户视野？

### F. 恢复与审计检查

- 这次改动若失败，是否仍能 fail-closed，而不是留下悬空状态、悬空审批或错误 owner？
- 是否存在对应的回归测试，证明对象边界、审批消费、恢复路径没有被打穿？

### G. 路书对齐检查

- 这次改动是在继续做硬 `P0 / 受控执行能力 + 任务状态机能力 + 高风险审批能力`，或补齐 `P1 / 恢复与渠道 contract`，而不是过早滑向体验花样或渠道特化？
- 这次改动是否让当前缺口更少，而不是只是把复杂性从 bridge 内部挪到用户面前？
