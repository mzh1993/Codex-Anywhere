# Reply Plane Origin-Manifest Design

日期：2026-04-04

状态：Phase 1 已落地；same-origin、manifest-only、相对 `cwd`、fail-closed 与原生 Feishu 产物回传已进入当前行为；explicit override 仍保留为 future

## 背景

当前 bridge 已经具备“状态卡片 + 最终文字摘要”回到当前 Feishu 会话的能力，但“用户可消费产物默认回当前会话”的 reply plane 仍未真正落地。

本设计只收口一个问题：

如何在不让 bridge 变厚、不增加用户心智的前提下，让最终摘要与最终可消费产物默认回到当前会话 origin。

## 已确认共识（一）

reply plane 不是“让 bridge 学会发东西”，而是“让最终可消费产物在 origin 约束下回到原会话”。

这里的 origin 不是 bridge 推理出来的语义，而是入站消息天然带来的地址事实，至少包含：

- `channel`
- `account`
- `conversation`
- `message`
- `thread`

bridge 不解释任务，也不拥有回传语义；它只执行一条硬规则：

- 如果用户没有显式改口，最终摘要和最终可消费产物只能回当前 `origin`

第一阶段纳入 reply plane 的对象只包括：

- 最终摘要
- 明确要回传的文件
- 明确生成的图片
- 明确生成的音频
- 明确生成的视频
- 明确形成的外部链接

第一阶段不纳入 reply plane 管辖范围的对象包括：

- 临时文件
- 原始命令输出
- 推理过程
- scratch 中间件
- 自动发现但未被声明的目录变化

这一段的核心原则是：

- 不是 bridge 决定“什么值得发”
- 而是系统只允许“被声明为最终产物的东西，按 origin 约束回原处”

## 已确认共识（二）

`delivery manifest` 不是给用户看的协议，而是 Codex 给 bridge 的最小投递清单。

它只应包含三类字段：

1. `summary`
   - 当前最终摘要
2. `deliverables`
   - 最终可消费产物数组
3. `note`
   - 产物的极短人类说明，可选

`deliverables` 第一阶段只允许以下 `kind`：

- `file`
- `image`
- `audio`
- `video`
- `link`

关键限制：

- manifest 不包含目标地址
- manifest 只接收 `Codex` 显式声明的最终产物；未声明即视为不存在
- manifest 不允许 bridge 再去扫描目录补猜
- manifest 不允许 bridge 通过目录 diff、工作区扫描或全文语义推断补出产物
- manifest 不允许混入中间产物
- manifest 不应成为新的用户交互协议

参考形状：

```json
{
  "summary": "已完成报告整理。",
  "deliverables": [
    { "kind": "file", "path": "reports/final-report.pdf", "note": "最终报告" },
    { "kind": "image", "path": "artifacts/chart.png", "note": "结果图表" },
    { "kind": "link", "url": "https://example.com/result", "note": "在线结果页" }
  ]
}
```

这意味着 bridge 的职责始终只有两步：

1. 校验这些产物是否合法、可发送、在允许范围内
2. 按当前 origin 发回去

因此，bridge 不是在“思考怎么发”，而是在执行 origin 约束和安全校验。

## 已确认共识（三）

用户不能感知 `delivery manifest` 的存在。

用户主路径保持不变：

- 用户仍然只发送自然语言
- Codex 内部决定是否形成最终产物声明
- bridge 不新增任何用户命令面
- Feishu 侧只呈现自然结果，而不是暴露 manifest 概念

表现形态应收口为：

- 没有最终产物时，只回摘要
- 有最终产物时，回摘要并把最终文件 / 图片 / 音视频 / 链接直接带回当前会话

因此，从用户感知上，系统只是“Codex 做完后把结果带回来了”，而不是“bridge 新增了一套产物协议”。

这也是本方案能保持北极星一致性的原因：

- 对用户零新增心智
- 对 Codex 是极薄格式约束
- 对 bridge 是纯执行，不拥有语义解释权

## 当前术语约束

为了避免 bridge 被表述成拥有“自己的心智”，本设计后续统一使用以下措辞：

- 不说“bridge 持有回传目标裁决权”
- 改说“bridge 执行 origin 约束”
- 不说“bridge 决定发到哪里”
- 改说“默认回传地址等于当前入站会话的 origin，只有用户显式改口才允许变化”

## 已确认共识（四）

第一阶段，本地产物路径最好只允许相对 `cwd`。

这不是为了追求最强功能，而是为了保持最稳、最薄、最不容易误传。

原因包括：

1. 它天然贴合当前任务边界
   - 当前任务本来就在某个 `cwd` 下执行
   - 若最终产物也必须位于这个边界内，则“它是不是本任务产物”有了最朴素的物理约束
2. 它避免 bridge 被迫理解宿主机语义
   - 一旦允许任意绝对路径，bridge 就需要开始面对“这是不是历史文件 / 别的任务产物 / 私人资料 / 系统敏感路径”等问题
   - 这会快速把 bridge 变厚
3. 它与 `larksuite/cli` 的安全方向一致
   - 该 CLI 对上传输入路径本身就倾向限制为当前工作目录内的相对路径
4. 它更符合“产物是任务内生成物”的用户直觉
   - 当前任务生成并准备回传的文件，默认应来自这次任务所在工作区，而不是宿主机某个角落的任意文件

因此，第一阶段建议强收为：

- `file`
- `image`
- `audio`
- `video`

这几类本地产物只接受相对 `cwd` 路径。

补充约束：

- 不接受绝对路径
- 不接受 `..` 越界
- 不接受解析后绕出 `cwd` 的符号链接
- `link` 类型单独使用 `url`，不走本地路径

## 已确认共识（五）

若 manifest 中某个产物不合法，bridge 必须 fail closed。

第一阶段的 fail closed 原则是：

- `summary` 照常回
- 合法产物照常回
- 非法产物不发
- 只补一条极短失败说明，不展开成长篇解释

第一阶段至少拦截以下情况：

- 文件不存在
- 路径不是相对 `cwd`
- 路径解析后越界
- 声明的 `kind` 与实际类型不匹配
- 文件体积超出当前 Feishu 投递能力
- 上传失败
- 当前渠道不支持该产物类型

期望的表现形态不是“整轮任务失败”，而是：

- 摘要正常回
- 成功的产物正常回
- 失败的产物在摘要后给一条短说明，例如：
  - `1 个产物未回传：路径越界`
  - `1 个产物未回传：文件不存在`
  - `1 个产物未回传：超过大小限制`

这条共识的核心意义是：

- 不因为一个附件失败就吞掉整轮结果
- 也不让 bridge 开始长篇解释“自己为什么判断失败”
- bridge 只是在不满足 origin / 边界 / 安全约束时拒绝执行对应投递动作

## 已确认共识（六）

同一轮里“摘要 + 多个产物”的 Feishu 呈现，第一阶段采用：

- 完成卡片承担结果锚点
- 原生 Feishu 消息承担产物交付

不采用“全部塞进一张结果卡”的方案，也不采用“只发产物不保留完成卡”的方案。

收口后的表现为：

- 运行中的同一张卡片，最终更新成“本轮结果”
- 结果卡只保留最终摘要
- 若有最终产物，则在同一会话 / 同一线程内继续发送原生产物消息

这条共识的核心口径是：

- 卡片负责总结
- 原生消息负责交付

这样可以：

- 保留 Feishu 原生文件 / 图片 / 音视频 / 链接的展示能力
- 避免 bridge 发明自己的“产物门户”“附件面板”或第二交互面
- 维持当前状态卡体系的连续性

## 已确认共识（七）

完成卡不做产物统计，只做结果锚点。

因此，第一阶段不建议在完成卡中增加如下噪音信息：

- `产物 3 个`
- `附件 2 个`
- `已回传 1 张图片`
- `成功 2 个，失败 1 个`

更贴北极星的收口方式是：

- 有产物，就直接在结果卡下方自然出现原生产物消息
- 没产物，就什么都不额外显示
- 只有发生产物回传失败时，才补一句极短失败说明

用户感知应收敛为：

- 任务完成了
- 如果有最终产物，它们自然跟在结果后面回来

而不是让用户先读一层“产物统计卡”再去对账。

## 已确认共识（八）

不同类型产物在第一阶段应尽量使用 Feishu 原生消息能力直接交付，而不是由 bridge 做统一抽象 UI。

建议收口如下：

- `file`
  - 直接回文件消息
- `image`
  - 直接回图片消息
- `audio`
  - 直接回音频消息
- `video`
  - 直接回视频消息
- `link`
  - 直接回一条极短文本 / 链接消息，可带极短 `note`

关键约束：

- `note` 默认关闭
- `note` 主要保留给 `link` 或“用户明确要求逐项说明”的场景
- `file / image / audio / video` 默认不附带额外 companion text
- 一旦某个产物已经成功发出，完成卡中就不再重复描述它
- 不在完成卡中再次抄写文件名、图片说明、音视频标题或链接地址
- `note` 不得重复文件名、链接地址或产物消息本身已经表达的信息
- 除非用户任务本身要求“逐项说明”，否则默认不做重复解释

这条共识的目标是避免形成这种低价值重复：

- 卡里先说“生成了 report.pdf”
- 下方文件消息又再展示一次 `report.pdf`

第一阶段应避免这种双重阅读负担。

## 已确认共识（九）

失败提示必须短到像一个“回传异常信号”，而不是一段解释文。

第一阶段建议只保留以下形态：

- `1 个产物未回传：文件不存在`
- `1 个产物未回传：路径越界`
- `1 个产物未回传：超过大小限制`
- `1 个产物未回传：当前渠道不支持`
- `1 个产物未回传：上传失败`

若存在多个失败，也不应展开成长列表；第一阶段建议最多只显示一条聚合短句，例如：

- `2 个产物未回传：路径或上传异常`

失败提示只回答两件事：

1. 有几个产物未回传
2. 属于哪类异常

失败提示不应暴露以下内容：

- 绝对路径
- 底层系统判断细节
- 安全规则触发细节
- 上传接口返回的原始错误
- 过长的排障说明

这些都不属于用户主视图，应留在内部日志、review 或后续运维排查面中处理。

这条共识的核心原则是：

- 让用户知道“有东西没回来”
- 但不让 bridge 借机长出解释世界的产品层

## 已确认共识（十）

合同矩阵层面，产品边界优先复用现有 `future` 行，不再为 reply plane 额外膨胀大量 `PB-*`。

当前可直接承接本设计方向的产品边界行包括：

- `PB-004`
- `PB-005`
- `PB-006`

其中已经覆盖：

- Feishu 作为闭环交互面
- 结果与用户可消费产物默认回原会话
- bridge 可以负责回传，但不能变成飞书业务助手或第二产品

因此，reply plane 第一阶段新增 `future` contract 时，优先补以下执行边界：

- `PM-005`
  - 本地产物仅允许相对 `cwd`
  - 越界或非法产物 fail closed
- `CT-004`
  - 同一任务的摘要回传与产物回传必须绑定同一 stored origin
- `OB-005`
  - 完成卡只保留摘要；产物原生回传；不显示产物统计；失败提示极短
- `XP-004`
  - `secure_linux` 与 `native_windows_fast` 必须遵守相同的 origin 约束和 manifest 语义

这条共识的核心原则是：

- 产品边界不再加词
- 执行边界补齐约束

## 已确认共识（十一）

`delivery manifest` 的最薄接入点只应包含两处：

1. `codex-exec` 提示词
2. `finishTask` 结束态解析

不新增：

- 新的用户命令面
- 新的 bridge 控制面
- bridge 主动目录扫描器
- bridge 语义猜测层

### 提示词接入

当前提示词已要求：

- `summary`
- `changed files`
- `next steps`

第一阶段的最小增量是：

- 保留 `summary`
- 增加可选 `deliverables`

也就是说：

- 没有最终可消费产物的任务，不需要输出 `deliverables`
- 只有确实存在最终可消费产物时，Codex 才声明它们

### 结束态接入

当前 `finishTask` 已抽取：

- `summary`
- `changedFiles`
- `nextSteps`

第一阶段只增加：

- `deliverables`

随后 bridge 只做三件事：

1. 解析 `deliverables`
2. 做边界校验
3. 按 stored origin 发回去

### 禁止扩厚

第一阶段明确不做以下行为：

- bridge 回头扫描 `cwd`
- bridge 用目录 diff 猜产物
- bridge 从全文语义推断“哪些文件值得发”
- bridge 自己决定某个文件是否属于最终交付物

### 失败处理

若 `deliverables` 缺失、格式错误、路径非法或上传失败：

- `summary` 仍正常返回
- 产物部分 fail closed
- 如确有必要，只补一条极短失败提示

这一条共识可压缩为一句话：

- manifest 只经过两只手：Codex 产出，bridge 执行

## 已确认共识（十二）

顶层需求仍保留一条未来能力：

- 若用户在当前任务中显式改口，系统后续可以偏离默认 `origin`

但在当前 `P0-收口 IV（反馈闭环阶段）`，reply plane 第一阶段不开放跨 origin 投递执行。

也就是说：

- 第一阶段唯一执行路径是 same-origin return
- “用户显式改口可偏离 origin” 目前只保留为后续能力保留位，不在本阶段落成运行时主路径

这样做的原因不是否认顶层需求，而是防止当前阶段为了支持少量特殊场景，重新把 bridge 做成：

- 自然语言目标解析器
- 临时跨渠道调度器
- 新的用户命令面

后续若要真正开放显式偏离 `origin`，必须同时满足：

1. 来源于当前任务中的用户显式表达
2. 具有结构化、可审计的 override 表达
3. 不新增新的 bridge 用户主命令面
4. 不允许 bridge 通过模糊自然语言自行猜测目标

在该能力尚未设计完成前，任何非 same-origin 投递都不属于 reply plane 第一阶段的默认闭环能力。

## 下一步待讨论问题

在当前 Phase 1 已落地基础上，后续继续讨论：

1. explicit override 应在哪个阶段独立立项，并如何保持“非 same-origin 不是默认主路径”
2. richer Feishu objects 是否继续沿“产物回传”而非“bridge 新控制面”这条线展开
3. README 与对外叙事何时同步跟上当前 reply plane 的已落地范围
