# V1 Closure Roadmap Design

日期：2026-04-04

状态：已确认方向，待按主线拆成实现计划

## 背景

当前仓库已经不是“能不能用”的问题，而是“怎样收口”的问题。

产品北极星没有变：

- 用飞书私聊直接指挥本地 Codex 干活
- 普通自然语言尽量直接给 Codex
- bridge 保持很薄
- 用户心智尽量低

因此，后续阶段不应继续无边界扩张，而应进入明确的 V1 收口模式。

## 问题

最近几轮推进里，仓库已经在很多细节上补得很深，但主线仍有三条没有彻底收口：

1. reply plane 还没有完全达到“稳定、克制、默认回原会话”的闭环状态
2. 持续任务稳态还没有完全达到“lane 连贯、状态可见但不烦”的长期可用状态
3. 泛化部署还没有完全达到“交给别人机器上的 Codex CLI 也能低心智装好”的程度

如果不明确排序，团队很容易重新陷回：

- 修很多细节
- 加很多局部能力
- 但主线迟迟没有真正封口

## 决策

### 一、后续只按三条主线收口

V1 后续只保留三条主线：

1. 回传闭环
2. 持续任务稳态
3. 泛化部署

除此之外，不再把零散体验细节单独当成一级目标。它们只能作为这三条主线的子问题存在。

### 二、三条主线必须串行，不并行发散

推荐顺序：

1. 先收回传闭环
2. 再收持续任务稳态
3. 最后收泛化部署

原因：

- 回传闭环直接决定“做完后结果有没有真正回到飞书”
- 持续任务稳态直接决定“每天用起来是否顺手”
- 泛化部署应建立在前两条主线已经稳定之后，否则只会把未收口语义更早扩散到更多机器

### 三、回传闭环的收口目标

回传闭环的核心目标不是“bridge 学会更多飞书动作”，而是：

- 当前私聊 / 线程发起的任务
- 在用户未显式指定别的去向时
- 最终摘要与用户真正可消费的产物
- 默认、稳定、克制地回到当前 origin

这里的“可消费产物”只包括：

- 最终摘要
- 明确要求回传的文件
- 明确生成的图片
- 明确生成的音频
- 明确生成的视频
- 明确形成的外部链接

不包括：

- 临时文件
- 原始命令输出
- 推理过程
- scratch 中间件
- bridge 通过扫描或猜测发现的变化

收口方向：

- bridge 只执行 same-origin 约束与最薄校验
- bridge 不猜产物，不扫描目录补产物
- 当用户只要一个主产物时，默认只回这个主产物，不顺手回伴随说明件
- 摘要必须稳定返回；产物失败不能吞掉摘要

### 四、持续任务稳态的收口目标

持续任务稳态的核心目标是：

- 同一任务 lane 连贯
- task id 不无故漂移
- bridge / gateway 重启后可自然续上
- 审批不会打断 continuity
- 执行中可观测，但不刷屏、不抖动

收口方向：

- task lane 是第一对象，单轮 run 只是 lane 上的一次执行
- `activeTaskId`、`lastTaskId`、`sessionId`、progress card 的关系必须持续一致
- 自然语言续写、审批后继续、重启恢复三条路径要统一落到同一 continuity 规则
- 长任务观测只保留最少状态集：启动、执行中、等待审批、完成、失败
- 执行中状态继续坚持同卡更新、文本去重、节奏收敛、长度稳定

### 五、泛化部署的收口目标

泛化部署的核心目标不是追求所有平台同等复杂能力，而是：

- 把当前仓库交给别人机器上的 Codex CLI
- 仍能以低心智方式装起并跑通主路径

第一阶段建议优先收口：

- Ubuntu 20.04 / 22.04
- WSL2 Ubuntu

原生 Windows 可以继续保留为受限能力或后续阶段，不把它作为当前收口阻塞点。

收口方向：

- 安装入口尽量统一成一段最短流程
- 默认目录、默认权限、默认运行时不要依赖机器专属硬编码
- preflight 与 `/codex doctor` 必须直接说明“缺什么、下一步怎么做”
- 运行时差异应收敛为少量 profile，而不是散落在文档和脚本缝隙里

### 六、后续所有推进必须遵守同一收口纪律

从现在开始，三条主线都使用同一推进顺序：

1. 先补顶层契约
2. 再补测试矩阵
3. 再改实现
4. 最后做真实灰度 replay

不能再采用“先修一个体验点，之后再补契约和回归”的临场式推进。

## 各主线的验收线

### 回传闭环验收线

- 用户要图，只回图，不顺手再回 md 说明
- 用户要文件，只回文件，不混入无关伴随件
- 摘要始终可回，产物失败不吞摘要
- 结果稳定回原私聊 / 线程，不因 profile 默认值、helper fallback 或 transport 噪声漂移

### 持续任务稳态验收线

- 同一任务不会无故切换 task id
- 服务重启后，下一句自然语言能续上原任务
- 审批前后仍处于同一 continuity lane
- 长任务过程中不再出现高频重复状态卡刷屏

### 泛化部署验收线

- 新机器按文档能完成安装、启动、doctor、首个任务
- 默认目录、默认权限、默认运行时符合设计预期
- 出问题时，doctor / preflight 能直接定位缺项，而不是靠人工猜

## 非目标

本收口阶段明确不做以下事情：

- 不把 bridge 扩成第二个产品
- 不新增桥自有的复杂命令面
- 不让 bridge 获得业务对象裁决权
- 不在收口阶段优先追求原生 Windows 完整体验
- 不把零散体验问题继续提升成新的一级 roadmap

## 预期结果

当这三条主线按顺序收口后，仓库应达到下面的状态：

- 用户几乎只需要在飞书里说话，或在必要时显式 `/codex ...`
- Codex 能持续、稳定地在同一条任务线上工作
- 做完后的摘要和主产物能默认回到当前会话
- 新机器部署不再依赖作者本人经验和环境记忆

一句话总结：

- 先把“回得准”收掉
- 再把“续得稳”收掉
- 最后把“装得广”收掉
