# Project North Star

> Internal decision note. This document is for product and engineering judgment, not public marketing copy.

## 一句话定义

- 对外用户语言：一个随时随地可用的远程 Codex 协作入口。
- 对内产品定义：一个以安全执行核心为底座的远程 Codex 协作桥梁。

## 第一原则

- 渠道不决定执行语义。
- 执行语义只能由状态机、审批协议、隔离边界与审计机制决定。
- Feishu、微信、未来任何入口，都只是 transport shell，不应改变任务如何开始、继续、审批、中断、恢复。

## 第一条红线

- 任何把“消息长什么样”直接等同于“系统该怎么执行”的设计，都是偏航。
- 任何依赖渠道猜测用户意图、跳过明确协议、绕开状态机的能力，都是偏航。

## 我们不做什么

- 不做一个更聪明的 IM 机器人。
- 不做一个靠提示词猜测一切的自动执行器。
- 不做一个渠道绑定型插件产品。
- 不让 Feishu、微信或 OpenClaw 反向定义执行核心。
- 不接受突破系统边界但规则模糊的产品定义。

## 当前阶段硬禁令

- 不让 bridge 接管普通任务语义；用户在“做什么、怎么做”上默认仍然是和 `Codex` 交互。
- 不把一次高风险批准扩成整段会话的长期权限；批准只作用于下一次获批 run。
- 不扩大 `bridge action` 劫持面；只有 repo-owned control-plane 能归 bridge，混合语义默认回落 `codex task`。
- 不靠“更聪明的猜测”减少审核或放松边界；bridge 优先做薄透传、边界守门、审批闭环、恢复与审计。

## 我们真正要做什么

- 把 `Codex` 封装成一个可远程进入的受控执行系统。
- 让用户不在电脑前时，仍能自然接续 Codex 的任务流。
- 让系统每一步都可解释：现在是什么状态、下一步允许做什么、为什么允许或拒绝。
- 让高风险动作进入明确审批协议，而不是混入普通对话流。
- 让中断、恢复、拒绝、失败都属于协议的一部分，而不是异常噪音。

## 本体与入口

- 本体：安全执行核心。
- 入口：远程 Codex 协作入口。
- 当前入口：Feishu。
- 未来入口：微信、其他 IM、Web 控制台、自动化系统。
- 入口可以替换；本体不能漂移。

## 主对象边界

- `codex task` 是用户真正连续协作的主对象。
- `bridge action` 是 bridge 为守住边界而临时接住的边界对象。
- 两者可以共享同一个对话入口，但不能共享执行语义和连续性状态。
- 默认让用户感知 `codex task`；只有在审批、边界拒绝、恢复提示、repo-owned control-plane 这类必要时刻，bridge 才短暂浮出。

## 控制面归属

- 本仓库专属宿主控制面动作属于 bridge，不属于普通 `codex task`。
- 这类动作的目标是维持桥本身的可用性与边界，例如 bridge / gateway 的健康检查、诊断、安装修复、服务控制。
- 这类动作必须纳入 bridge 的审批 / 审计 / 隔离治理：只读查询可直接执行，变更类动作先审批后执行；且始终与普通任务连续性分开。
- 同一个入口可以同时承载工作任务与控制面动作，但不能让后者污染前者。

## 当前最小产品边界

- 文本输入
- 任务状态
- 高风险审批
- 中断恢复
- 边界拒绝
- 基础审计

## 需求裁决五问

- 它是否让渠道开始决定执行语义？
- 它是否削弱了状态机的明确性？
- 它是否绕过了审批或边界？
- 它是否破坏了任务连续性？
- 它是否增加了用户理解系统行为的不确定性？

只要有一个答案是“是”，默认不做。

## 我们希望用户感受到什么

- 不是“我在跟一个机器人碰运气”。
- 而是“我随时都能接上我的 Codex 工作流，而且系统行为始终可预期”。

## 终局能力图

### 用户最终感知的 3 个核心价值

- 随时接入：不在电脑前，也能自然接上 Codex 工作流。
- 连续协作：任务不会因为渠道、会话、中断而失去上下文和边界。
- 可控放心：高风险动作有明确边界、审批和审计，不靠碰运气。

### 支撑这 3 个价值的 5 个核心能力

- 受控执行能力：决定什么可执行、什么需审批、什么直接拒绝。
- 任务状态机能力：决定任务如何开始、继续、中断、等待输入、等待审批、完成、终止。
- 恢复与审计能力：决定异常之后如何回收、继续、追踪。
- 渠道解耦能力：决定 Feishu、微信、Web 只能当入口，不能接管执行语义。
- 协作体验编排能力：决定文本、状态、审批、图片、中间结果如何组成低负担体验。

### 能力层次

- 本体层：`受控执行能力 + 任务状态机能力 + 恢复与审计能力`
- 结构保障层：`渠道解耦能力`
- 用户感知层：`协作体验编排能力`

### 当前项目位置

- 已有最小闭环：`文本 + 状态 + 审批`
- 已有受控 Runner 雏形：允许、审批、拒绝、恢复、继续 已基本成形
- 当前仍是 V1：可用，但远未到优秀体验与强边界产品化阶段

### 当前最大缺口

- 受控执行能力不够硬：还需要从提示词匹配升级到执行边界约束。
- 状态机表达不够统一：协议已成形，但还没完全做到低歧义、低认知负担。
- 恢复与审计不够产品化：现在更像工程能力，还不是完整体验能力。
- 渠道解耦还停留在理念和局部实现：还没抽成真正稳定的 channel contract。
- 协作体验层还很早期：远未展开图片、中间结果、结构化状态、流程视图。

### 决策优先顺序

- 先守住 `受控执行能力 + 任务状态机能力`
- 再补强 `恢复与审计能力 + 渠道解耦能力`
- 最后放大 `协作体验编排能力`

## 终局愿景

- 不管人在哪里、用什么渠道，只要有网络，就能以一致、连续、可控的方式协作 Codex，既能做复杂开发任务，也能完成简单电脑操作，同时不丢失边界感与可审计性。
